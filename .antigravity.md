# Ability Hub Project Log

## Context
Ability Hub is a management platform for learners, operators, and appointments. It uses Laravel, Blade, AlpineJS, and Tailwind CSS.

## Key Considerations
- Sorting functionality is implemented using query parameters `sort` and `sort_order`.
- Controllers (`LearnerController`, `OperatorController`) handle the sorting logic and should always pass `$sort` and `$sort_order` to the views.
- UI uses class-based Blade components and partials.
- Z-index and `overflow` properties on parent containers can interfere with absolute-positioned elements like dropdowns.

## Fixed Issues & Errors
### 1. Broken 'Sort by' Dropdown (2026-03-26)
- **Problem:** The dropdown was not showing fields or not opening correctly in learners and operators lists.
- **Cause:** 
    - `overflow-hidden` on parent cards clipped the absolute-positioned dropdown.
    - `overflow-x-auto` on the top bar in learners list also clipped the dropdown.
    - `z-index` (z-10) was potentially too low.
    - Logic for determining the current sort in the partial was using raw request data inconsistently.
    - `x-cloak` was used without CSS definition (initial flicker or hide issues).
- **Fix:**
    - Removed `overflow-hidden` from main index cards.
    - Removed `overflow-x-auto` from the top bar in `learners/partials/list.blade.php`.
    - Increased `z-index` to `z-50` for dropdowns.
    - Added `[x-cloak] { display: none !important; }` to `app.css`.
    - Improved the partials to show the current selection on the button and use a unified mapping for labels.
    - Fixed a potential `Undefined index` bug in controllers when `sort_order` was missing from the request.
- Added `[x-cloak] { display: none !important; }` to `app.css`.

### 2. Redesign of Learner/Operator Forms & Availabilities (2026-03-26)
- **Problem:** Forms and availability panels were "horrible" and not mobile-friendly.
- **Fix:** 
    - Implemented modern tab switchers.
    - Grouped fields into logical sections with icons.
    - Replaced table-based availability grid with a responsive day-card grid.
    - Added `ResourceHeaderCard` to edit pages for better context.
    - Ensured consistency with "sober but lively" style (rounded corners, subtle shadows, etc.).

### 3. One-Hour Shift in Availability Slots (2026-03-26)
- **Problem:** Availability slots in `scatter-plot-week.blade.php` were shown with a one-hour offset (e.g., 10:00 instead of 09:00).
- **Cause:** Slots are stored as wall-clock hours/minutes in the DB, but were being converted to UTC Carbon objects and then localized by JS. The 'Z' suffix in the ISO string led the browser to shift the time by the local timezone offset.
- **Fix:** Changed `toIso8601String()` to `toDateTimeLocalString()` in `scatter-plot-week.blade.php` to provide an offset-free ISO string, which browsers treat as local time.


## Project Rules
- Follow PSR-12 for PHP.
- Use UUIDs and traits consistently.
- Keep controllers thin; delegate to Actions and Services.
- Use Blade components + AlpineJS + Tailwind for frontend.
- Maintain localization (Italian).
- Ensure accessibility.
- Support three channels: Blade, API, MCP.
- **Always translate all UI strings in `lang/it.json` and keep it updated.**

### 4. Lentezza Caricamento Appuntamenti (2026-04-03)
- **Problema:** La rotta `/appointments` era estremamente lenta (4s+ in produzione) e faceva 3 richieste API identiche simultanee all'avvio.
- **Cause:**
    - **N+1 Problem:** Nel modello `Appointment.php`, il metodo `toFullCalendar()` accedeva alle relazioni `learners` e `operators` (molti-a-molti), ma il controller caricava solo le versioni singole (`learner`, `operator`). Questo generava centinaia di query extra per caricare le collezioni.
    - **Alpine Initialization Loop:** L'uso di `x-init="init()"` insieme a una funzione nominata `init()` nell'oggetto data (Alpine 3 auto-chiama `init()`) causava una doppia inizializzazione. Il watcher su `currentWeekStart` (attivato durante l'assegnazione nel costruttore) aggiungeva una terza chiamata concorrente.
    - **Mancanza Indici:** Mancavano indici sulla colonna `starts_at` nella tabella `appointments`, obbligando il database a scansioni complete durante i filtraggi per range.
    - **Payload API Ridondante:** Ogni richiesta API includeva tutti gli operatori, learner, discipline e tipi di appuntamento (meta-data), appesantendo inutilmente il payload dopo il primo caricamento.
- **Fix:**
    - **Backend:** Eager loading delle relazioni `learners`, `operators` e `appointmentType` nel `ListAppointmentsController` per risolvere l'N+1.
    - **Frontend:** Rimosso `x-init` ridondante e implementato un sistema di deduplicazione via `activeRequests` (promise cache) per bloccare richieste simultanee per lo stesso mese.
    - **Protocollo:** Introdotto parametro `include_meta=0` per caricare i dati accessori solo sul primo fetch o in caso di force-refresh.
    - **Database:** Aggiunta migrazione con indici su `starts_at` e composito `(user_id, starts_at)`.
